# Admin Usage Platform Requirements

请把当前 `admin-usage` 从“轻量请求日志后台”升级成“可视化产品数据平台”。

## 当前问题

现在这套系统只能看到基础请求统计，比如请求数、访客数、耗时、热门接口、最近请求。它更像接口访问日志面板，不像产品数据平台。缺少真正的数据可视化，也缺少和 La vérité 核心业务结果相关的数据分析能力，无法回答“用户在验证什么、系统给出了什么结果、哪里经常失败、趋势怎么变化”这几个关键问题。

## 目标

下一版后台需要面向“产品分析 + 系统监控”双目标建设，而不是只做代理请求统计。要让我进入后台后，第一眼就能看到趋势、结构、异常和重点问题，而不是只能翻表格。

## 必须优先补的内容

### 1. 可视化图表

需要加入真正的 dashboard 图表，而不是只保留表格。优先增加：

- 每日 / 每小时验证次数趋势
- 平均分数趋势
- 分数区间分布
- verdict 分布
- claim type 分布
- impact 分布
- 渠道命中分布
- connector error 分布
- 请求耗时趋势和 P95 / P99

### 2. 业务事件埋点增强

当前事件字段太少，只记录了输入长度、媒体数量、type、impact，不足以支撑业务分析。请在不保存用户全文的前提下，为 `/api/check` 增加结果摘要字段，建议至少包括：

- `finalScore`
- `verdict`
- `cap.note`
- `supportCount`
- `refuteCount`
- `channelHitCount`
- `claimCount`
- `hasMedia`
- `hasAcademicEvidence`
- `aiCommitteeEnabled`
- `retrievalStageCount`
- `connectorErrors`
- `elapsedMs`

### 3. 产品视角指标

后台必须能直接看到这些核心业务指标：

- 总验证次数
- UV / PV
- 平均最终分数
- 高可信 / 中可信 / 低可信占比
- 各类 claim 占比
- 高频封顶原因
- 高频风险项
- 最常失败的连接器
- 最耗时的验证请求
- 最近是否出现大面积低分 / 高错误率

### 4. 筛选与钻取

不能只给总览。需要支持按这些维度筛选：

- 时间范围
- type
- impact
- verdict
- 分数区间
- 是否带媒体
- 是否触发学术渠道
- 是否出现反证
- 是否出现 connector error
- 是否启用 AI committee

同时，希望图表或指标卡可以点进去，看对应明细。

### 5. 页面结构重做

现在的页面结构还是“几个 KPI + 两张表”，信息层级不够。建议改成：

- 顶部：核心指标卡
- 中部：趋势图、分布图、结构图
- 下部：错误监控、渠道监控、性能监控
- 底部：明细事件表

也就是先做 dashboard，再做 detail table。

### 6. 监控板块独立化

系统稳定性不能埋在“最近请求”里。请单独增加：

- 错误趋势
- 状态码分布
- connector failure top list
- 代理层 502 统计
- 上游服务异常统计
- 平均耗时 / P95 / P99

这样后台才能同时承担运维监控作用。

### 7. 噪音过滤

像 `/favicon.ico`、静态资源请求这类噪音事件不要污染核心统计。请加过滤规则，至少默认不纳入业务分析指标。

### 8. 存储层考虑升级

当前 NDJSON 适合快速验证，但如果事件量增加，统计会越来越慢。短期可以继续保留 NDJSON；中期建议考虑 SQLite 或 DuckDB，把聚合分析能力做实，否则这个后台很快会成为瓶颈。

### 9. 部署和权限

如果这套后台未来要上线，不能默认裸奔。请补：

- 默认认证机制
- 更完整的环境变量说明
- 部署说明
- 数据文件位置和备份建议

## 我对下一版的预期

不是继续把日志堆得更多，而是做成一个真正能回答以下问题的后台：

- 用户最近都在验证什么类型的信息？
- 系统最近整体分数分布如何？
- 哪些风险项和封顶原因最常出现？
- 哪些连接器经常失败？
- 请求量、耗时和错误率有没有恶化？
- 哪些验证链路最常被用户使用？

## 结论

当前版本可以保留，作为 V0 的旁路日志后台；但下一版请按“产品数据看板 + 系统监控后台”的方向重构，不要继续停留在轻量日志页面层面。
